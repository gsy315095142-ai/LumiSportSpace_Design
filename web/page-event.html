📐 文档 6 步 vs 代码 4 态

产品设计定义 6 个流程阶段，当前代码实现为 4 个 SessionStatus。部分阶段在代码中共享同一状态，需后续开发独立拆分。

文档 6 步 代码 SessionStatus 代码区分方式 开发状态
① ⚙️ 玩法配置 waiting 管理员尚未点「确认玩法」与②共用同一状态 ✅ 已实现
② ⏳ 等待报名 waiting 管理员已点「确认玩法」，尚未「开始报名」与①共用同一状态 🟡 子状态待独立
③ 🤺 报名中 betting guessingOpen = false选手可报名，观众不可下注 ✅ 已实现
④ 🎰 竞猜中 betting guessingOpen = true管理员点「开启竞猜」后观众可下注 🟡 子状态待独立
⑤ ⚔️ 比赛中 started — ✅ 已实现
⑥ 🏆 已结算 settled — ✅ 已实现

⚠️ 子状态待独立说明

①② 合并为 waiting：当前靠管理员是否已点「确认玩法」来区分，但代码中无独立字段标记。
③④ 合并为 betting：通过布尔字段 guessingOpen 区分，已可正常工作，但若需前端根据步骤展示不同 UI（如进度条高亮第 3 步 vs 第 4 步），建议后续将 guessingOpen 提升为独立的 SessionStatus 枚举值。

🖥️ Server 端三大界面

Unity 大屏在赛事不同阶段展示三个核心界面，与小程序 6 步流程对应关系如下：

01
Server 端首页：管理员选择竞技玩法
对应 ① 玩法配置 对应 ② 等待报名

管理员在 Server 端选择游戏类型（冰球/拳王/击剑）并确认，小程序端同步进入等待状态。待开发：Server 端模式切换将统一收归到首页（当前在匹配页面）。

02
Server 端匹配页面：选手就位与竞猜
对应 ③ 报名中 对应 ④ 竞猜中 对应 ⑤ 比赛中

小程序端进行报名参赛、竞猜胜负；管理员审核报名、开启竞猜、开始比赛。Server 端展示匹配进度与比赛实况。

03
Server 端结算页面：胜负与下一局
对应 ⑥ 已结算

展示比赛结果与积分变动。管理员可选择再来一局（继承玩法配置）、切换游戏或返回首页（收起当前场次）。

📱 小程序管理模式 默认

PC 操控开关 = 关闭时，仅小程序管理端可操控赛事流程（配置、报名、竞猜、开赛、结算等）。Server 端仅展示画面，不接受操控指令。

🖥️ PC 控制台模式 需手动开启

PC 操控开关 = 开启时，仅 PC / Server 端可操控赛事流程。此时小程序管理端的操控按钮将禁用，仅保留状态查看功能。

两端 互斥控制 · 同一时刻仅一端可操控，状态数据始终实时同步
